# Contributing to HAL YieldOS

Thank you for your interest in contributing.

---

## Project Boundary

HAL YieldOS is a **read-only Functional Yield Evidence Layer**. Every contribution must preserve this boundary.

All contributions must maintain:

- `read_only` behavior — no hardware state changes
- `candidate_only` outputs — no certified decisions
- `human_review_required` — no autonomous action
- No hardware control of any kind
- No recipe control or modification
- No autonomous recovery execution
- No safety certification claims
- No yield guarantee claims
- No root-cause certification claims

If a proposed change would cause YieldOS to control hardware, modify recipes, execute recovery profiles, or make safety/yield/RCA certifications — it will not be accepted.

---

## Welcome Contributions

### Tests

- Additional unit tests for existing analyzers
- Edge case coverage for confidence scoring
- Sample data variation tests
- Strict validation contract tests
- Domain boundary tests

### Documentation

- Clarifications to safety boundary documentation
- Demo guide improvements
- Sample output explanations
- Typo fixes and clarity improvements
- Translated documentation (Japanese and Korean already exist in `docs/ja/`)

### Sample Data

- Synthetic or fully sanitized sample datasets
- Additional domain scenarios (robot, semiconductor, space, memory)
- New fault patterns for existing analyzers
- No real customer or proprietary data

### Schema and Evidence Clarity

- Improved evidence lineage documentation
- Better missing evidence explanations
- Clearer valid-condition descriptions
- Report readability improvements

### Validation Improvements

- Stricter contract validation checks
- Better error messages for missing evidence
- Improved `yieldos validate --strict` coverage

---

## Not Accepted

The following contributions will **not** be accepted:

- Robot command execution (ROS, motion, torque, joint)
- Semiconductor recipe modification or equipment control
- Hardware runtime profile application
- Firmware flashing or device write operations
- Production safety certification claims
- Root-cause certification claims
- Yield guarantee claims
- External cloud or API dependencies (without explicit discussion)
- Autonomous recovery execution
- Timing closure claims
- Any change that sets `hardware_execution_enabled = true`

## Before Submitting

1. Run the full test suite:

```bash
python -m pytest -q
```

2. Run the health check and demo:

```bash
yieldos doctor --deep
yieldos demo --all --out output/demo_all
```

3. Validate a case strictly:

```bash
yieldos validate --case output/demo_all/semiconductor --strict
```

4. If applicable, run the ruff linter:

```bash
python -m ruff check .
```

5. Make sure `recovery_profile.json` is not generated by your change.

---

## Pull Request Guidelines

- Use the pull request template (`.github/PULL_REQUEST_TEMPLATE.md`).
- Keep changes focused — one concern per PR.
- Include test coverage for new behavior.
- Do not commit generated `output/`, `dist/`, or `build/` directories.
- Do not include real production or customer data.

---

## Reporting Issues

Use the GitHub issue templates in `.github/ISSUE_TEMPLATE/`:

- `bug_report.md` — for bugs and unexpected behavior
- `sample_data_request.md` — for sample data schema questions
- `documentation.md` — for documentation improvements
